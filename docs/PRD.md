# Product Requirements Document — AI Agent Debate System
## Exercise 02: Orchestration of AI

---

## 1. Problem Statement

AI orchestration frameworks require a concrete, testable scenario to evaluate multi-agent coordination.
A structured debate is ideal: it demands clear turn-taking, message routing, role specialisation, and
a definitive resolution mechanism — all under a single supervising entity.

---

## 2. Goals

| # | Goal | Success Criterion |
|---|------|------------------|
| G1 | Real AI debate | All arguments generated by live Claude API calls, not hardcoded text |
| G2 | Clear hierarchy | One parent agent (Judge) spawns and supervises two child agents (Pro, Con) |
| G3 | No direct child-to-child comms | Pro and Con never hold references to each other |
| G4 | Mandatory skills | Seven named Skills fully implemented, loaded, and active |
| G5 | Internet-backed evidence | Every argument cites a real web source retrieved via tool_use |
| G6 | Definitive verdict | Judge declares a winner; ties are structurally forbidden |
| G7 | Budget control | Gatekeeper enforces a per-session USD cap |
| G8 | Resilience | Watchdog detects stalled calls; Gatekeeper retries transient failures |
| G9 | Observable | All exchanges written to a structured JSONL log |
| G10 | Testable offline | Full test suite runs without real API keys |

---

## 3. Non-Goals

- Multi-model provider support (only Anthropic is required)
- Persistent debate storage / database
- Web UI (terminal CLI is sufficient)
- Real-time streaming of agent responses

---

## 4. User Stories

**As a student/operator:**
- I can start a debate from a terminal menu without editing any source files.
- I can change the topic and round count from the same menu.
- I can read the full debate transcript in the terminal immediately after it ends.
- I can re-run the system with `python main.py --run` from a CI script.

**As a developer:**
- I can import `DebateRunner` and call `runner.run()` to get a typed transcript and verdict.
- I can add a new Skill by writing a `SkillDefinition` and registering it — no agent code changes.
- I can run `uv run pytest` offline to validate all logic.

---

## 5. Debate Mechanics

| Parameter | Default | Configurable |
|-----------|---------|-------------|
| Topic | "Artificial Intelligence will do more good than harm for humanity" | `config/debate.yaml` or `DEBATE_TOPIC` env var |
| Rounds per side | 5 (budget mode) | `config/debate.yaml` or `DEBATE_ROUNDS` env var |
| Model | claude-haiku-4-5-20251001 | `config/models.yaml` |
| Budget cap | $2.00 | `config/debate.yaml` |

**Round flow:**
1. Judge introduces round.
2. Pro generates argument or rebuttal (may call web search tool).
3. Judge routes Pro's message to Con as context.
4. Con generates argument or rebuttal.
5. Judge summarises the round.
6. Repeat for N rounds.
7. Judge renders scored verdict — one winner, no ties.

---

## 6. Skills Specification

Each skill is a `SkillDefinition` Pydantic object registered in `SkillRegistry`.
Skill `instructions` are injected into the owning agent's system prompt before each session.

| Skill | Owner | Purpose |
|-------|-------|---------|
| `judge_moderation_skill` | Judge | Neutral moderation, round intros, summaries |
| `pro_argument_skill` | Pro | Structured pro-side argument construction |
| `con_argument_skill` | Con | Structured con-side argument construction |
| `evidence_retrieval_skill` | Pro, Con | Web-search-backed evidence sourcing |
| `rebuttal_skill` | Pro, Con | Direct counter to opponent's last argument |
| `verdict_skill` | Judge | Scored final judgment, no ties allowed |
| `json_protocol_skill` | All | Enforce raw JSON as the only output format |

---

## 7. Quality Constraints

- **Persuasion over fact**: the Judge scores persuasiveness, not just factual accuracy.
- **Responsiveness**: agents that ignore the opponent's argument score lower.
- **Respectful tone**: both agents are instructed to be civil; the Judge may note any violations.
- **No repetition**: agents are instructed not to restate arguments they have already made.
